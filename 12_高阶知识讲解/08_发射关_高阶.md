# 高阶讲解 · 作品集策展策略与 AI 系统设计面试的标准追问顺序

## 定位

自学阶段学的是"整理 GitHub、写简历、准备路演"。这段要讲清楚:**发射关不是"把之前做的东西打包一下",是用招聘方的眼光重新审视 8 关积累的所有材料,决定"哪些留、哪些砍、哪些要重新讲一遍故事"。**

## 一、作品集策展:精选 4 个,不是堆砌 8 个

8 关下来学生手上有 8 个 GitHub repo,但简历和 profile 只能放得下 3–4 个精品,策展逻辑:
1. **深度优先于数量**——一个从第 3 关滚动到第 7 关、完整走过评测和成本核算的项目,远胜过 8 个都停留在"跑通了"阶段的项目(参见 `10_企业场景案例库/场景选题矩阵` 的"滚动升级"设计);
2. **四件套完整度是硬指标**——README(架构图+demo)、evals/(测试集+跑分)、COST.md(成本核算)、POSTMORTEM.md(踩坑复盘),四样不全的项目不进精选;
3. **和目标岗位对齐**——按《AI 岗位能力地图》,精选的项目要能覆盖目标岗位最看重的 2–3 项能力,而不是平均用力。

## 二、"招聘官 10 秒动线"——README 的专业写法

招聘方看一个 repo 的 README 平均只给几十秒,策展要求:
- 第一屏必须有:一句话说清楚这个项目解决什么问题 + 一张能一眼看懂的架构图或 demo 动图;
- 第二屏:量化结果(评测跑分、成本数字、召回率提升百分比)——不要用"效果不错"这种话,要有具体数字;
- 往下才是技术细节和安装说明——技术细节永远不该出现在第一屏。

## 三、AI 系统设计面试的标准追问顺序(比背题库更重要的是理解这个套路)

企业面试官对 AI 应用类项目的追问,几乎都遵循同一个套路(和 `05_就业面试/面试官人格库与追问题库` 的四类面试官对应):
1. **业务层**:这个项目给谁用?解决了什么真实痛点?为什么是你做这个而不是别人?
2. **架构层**:技术选型的取舍是什么?为什么用 RAG 而不是微调,为什么用这个检索策略?
3. **工程层**:成本多少、延迟多少、怎么监控、挂了怎么办、怎么回滚?
4. **评测层**:你怎么知道这个系统"好"?评测集怎么来的,LLM judge 校准过没有?
5. **安全层**:给我一个攻破你系统的例子,你怎么防?

**教学价值**:与其死记硬背题库,不如理解这套"业务→架构→工程→评测→安全"的固定追问深度顺序——任何一个环节答不出来,面试官就会在那一层反复深挖,这也是为什么八关任务卡从第 1 关到第 7 关的顺序本身就是按这套追问逻辑设计的。

## 四、常见误区

1. **只准备了"怎么讲优点",没准备"怎么讲失败"**——面试官最想听真实的踩坑故事,POSTMORTEM.md 里的内容是面试的核心素材,不是可有可无的附件;
2. **路演只讲功能演示,不讲评测数据**——参见第 7 关铁律"跑通了不算完成,评测过了才算",路演同理;
3. **简历项目描述用形容词堆砌("高效的""智能的"),不用量化数字**——面试官对形容词免疫,只对数字和具体机制有反应。

## 五、来源锚点

- 2026 届校招简历筛选标准调研:GitHub 上"干净 repo + eval traces + 成本记录 + 失败复盘"优于花哨的 side project;GitHub 高星/开源参与可跳过初筛环节——具体调研见教师记录,建议每学期开课前重新核验。

## 六、本关实战钩子(直接改验收 rubric)

- 精选作品集必须是 3–4 个而不是全部 8 个,并附一段"为什么精选这几个"的策展说明;
- 每个精选项目的 README 必须通过"10 秒动线"检查(第一屏有一句话定位+架构图/demo,不能第一屏就是安装说明);
- 模拟终面环节,学生必须能按"业务→架构→工程→评测→安全"五层顺序,针对自己精选项目逐层作答,答不上的层次记为待补强项。
