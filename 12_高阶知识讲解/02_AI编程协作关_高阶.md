# 高阶讲解 · 规格驱动开发与 AI 编程工具的正确分工

## 定位

自学阶段学的是"怎么用 Claude Code/Cursor 生成代码"。这段要讲清楚:2026 年企业 JD 原文写的是"深度使用 AI 编程工具、规格驱动开发",不是"会点自动补全"——**AI 编程能力已经从"效率工具"变成岗位硬性要求**,面试会具体问你的协作模式而不只是最终代码。

## 一、行业共识:两类工具不是竞争,是互补的认知模式

2026 年的框架已经从"选哪个工具"变成"什么任务配什么工具":Cursor 押注 IDE(fork 了 VS Code,在编辑器里做 agent mode);Claude Code 押注终端(把整个仓库当工作区的 CLI,把 hooks、skills、子代理、MCP 都做成一等公民)。**主流工作流**:Cursor 留在编辑器里做小范围紧密编辑(Tab 自动补全、行内编辑);遇到大任务(重构、迁移、多包改动)时,在旁开一个终端跑 Claude Code,描述目标结果,继续在 Cursor 里干别的活,Claude Code 跑完(5–60分钟)再回来审 diff。

**安全实践**:两个工具不要同时改同一个文件(会互相覆盖对方的自动保存);关掉不想被意外触发的自动执行 shell 命令功能;每次 agentic 跑批之间要 commit,保证随时能回滚到干净状态。

## 二、规格驱动开发(Spec-Driven Development)

**核心问题**:静态的规格文档几小时内就会和实现脱节,所以"规格生命周期怎么运转"是采用任何规格驱动工具前第一个要回答的问题。

代表性工具生态:
- **BMAD-METHOD**(开源,4.8万★):用命名角色(业务分析师/产品经理/架构师/开发/QA)编排多个专职 Agent,靠"文件化上下文传递"和严格角色边界完成全 SDLC 流程;缺点是协调开销大——实现阶段暴露设计问题时,每次交接都要重新走文件传递;
- **Cursor 的 `.cursor/rules/*.mdc`**:更轻量,用 Markdown+YAML frontmatter 做四种激活模式(始终生效/自动附加/按需请求/手动),持久化项目级 AI 指导;实践中规则数量一多,没设成"始终生效"的规则容易被 Agent 漏掉。

**教学落地**:第 2 关的固定任务"用 Claude Code/Cursor 完成一次规格驱动开发",不是让学生随便写个 prompt 让 AI 生成代码,而是要求产出 **spec → 代码 → 测试** 全链路,并且能讲清楚"这份 spec 后来和实现出现了什么偏差、怎么处理的"。

## 三、Claude Code 的子代理(Subagent)机制——理解"为什么"比"怎么用"更重要

子代理是专职处理特定任务类型的 AI 助手,每个都跑在自己独立的上下文窗口里,有独立的系统提示词、工具权限和许可——这既是**约束手段**(限制工具访问范围)也是**成本控制手段**(把任务路由给更快更便宜的模型,比如 Haiku)。这和第 3 关讲的"上下文隔离(Isolate)"策略是同一件事的两种说法——AI 编程工具本身就是上下文工程原理的产品化。

## 四、常见误区

1. **把 AI 编程等同于"复制粘贴 AI 生成的代码"**——企业要的是"能设计可复用 Skill、能编排多 Agent 协作系统"的深度使用能力;
2. **不做版本控制习惯**——agentic 跑批之间不 commit,出问题无法回滚;
3. **规格写完就不管了**——不建立"规格 vs 实现"的对齐机制,规格文档几小时内作废。

## 六、本关实战钩子(直接改验收 rubric)

- 提交物必须包含 spec 文档(哪怕只有一页)、对应的代码 commit、以及"spec 和最终实现的偏差说明";
- 至少展示一次"大任务交给 Agent 跑、自己同时处理别的事"的协作证据(commit 时间线可以体现);
- 硕士生加做:对比"直接让 AI 一次性生成完整功能" vs "拆成 spec→分步实现→测试"两种方式,在正确率和返工次数上做量化对比。
