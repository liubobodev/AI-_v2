# 第 3 关 · Prompt 资产关(W5–6)

## 目标
做一个真的有人愿意用的 AI 小产品,并把 Prompt 从"一次性咒语"变成**有版本、有评测的工程资产**。

## JD 原文对标
> "建设 **Prompt 模板与版本管理**、评测集与自动化回归……等应用层通用能力"——2026 届 LLM 应用开发工程师 JD

## 技术栈
任一模型 API、结构化输出(JSON Schema)、git 管理 prompt 模板。

## 任务
1. 从你的赛道里选一个单点任务(如:评论情感抽取、合同条款提取、教案生成)
2. Prompt 写成**模板文件**进 git:角色/任务/约束/few-shot 示例分离,变量用占位符
3. 输出必须是**结构化 JSON**(schema 校验,不是"求模型输出 JSON")
4. 建 20 条评测样本(输入+期望输出),写脚本跑准确率
5. 至少迭代 3 个版本,每版记录:改了什么 → 准确率变化(这就是最小的 prompt 回归)

## 验收 rubric
- [ ] prompt 是文件不是截图,git 里 ≥3 个版本
- [ ] 输出 100% 通过 schema 校验
- [ ] 评测脚本可一键复跑,3 版准确率有记录(如 65%→80%→87%)
- [ ] 有 1 个"改了反而变差"的版本及原因分析(没有 = 迭代不够)

## 简历句模板
> 构建 XX 结构化抽取产品:Prompt 模板化+版本管理,自建 20 条评测集驱动迭代,准确率 X%→Y%

## 面试追问预演
- 为什么这里用 few-shot 而不是 fine-tune?
- 模型升级了你的 prompt 全要重调吗?(考:评测集就是护城河)

## 双工具用法
- **NotebookLM**:上传你的 3 版 prompt + 评测记录 → 用 **Critique 模式音频**听 AI 批评你的设计 → 记录反驳点
- **Obsidian**:评测结果表用 Bases 记录(属性:版本/准确率/改动点),自动生成迭代曲线素材
