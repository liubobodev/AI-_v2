# AI 应用开发核心能力地图 · 五层体系

## 为什么要有这张图

八关是**任务驱动的纵切面**——按项目产出排关卡顺序。这张图是**专业驱动的横切面**——按"AI 应用开发工程师"这个岗位的完整能力结构分层。两者互补:八关回答"这周该做什么",这张图回答"我在整个专业地图里站在哪、还缺什么"。

多方 2026 年 AI Engineer 能力框架交叉验证后,行业共识收敛到一条学习顺序:**Python/REST API → LLM API 与提示词 → RAG 与向量库 → Agent 与编排 → MCP 与工具集成 → 部署与 LLMOps → 作品集**,且明确指出"评测是区分 AI 工程师和 AI 爱好者的最强信号"——这和本训练营"Evals 是第一硬通货"的设计完全吻合,说明八关的骨架是对的,**缺的是横向补全**。

## 五层能力模型

### L0 基础层(课前自学,不占八关课时)

Python 工程能力(不止脚本,要有函数/类型注解/虚拟环境/async-await)、REST API 与 JSON、机器学习基本概念(过拟合、训练/验证/测试集划分)、神经网络与 Transformer 直觉。**这层已经在《AI应用开发课程》L0 模块(独立教研产出)覆盖,本训练营默认学生已具备或课前自学补齐。**

### L2 工程化层(部分覆盖,需要加深)

- **LLMOps 全链路**:Prompt 版本注册表、成本/延迟全程追踪、golden dataset 治理——第 7 关已有评测和成本表,但没有系统讲清楚 Langfuse(可观测追踪)/Helicone(成本追踪)/LangSmith(Prompt 注册与团队协作评测)这几类工具各自的分工边界;
- **框架选型判断力**:LangGraph(生产级、失败代价高的系统首选)vs CrewAI(快速业务流程自动化)vs PydanticAI(强类型保证)vs 手写循环——2026 年的框架格局已经分化清楚,但本训练营目前没有专门讲"什么时候该用框架、什么时候手写更好"。

### L3 决策层(完全空白,本次新增)

**这是一线企业默认候选人具备、但培训班几乎不教的判断力**:

- **模型选型与决策框架**(新增 → `01_模型选型与决策框架`):Prompt→RAG→Fine-tune→Distill 的顺序判断,LoRA/QLoRA,"大部分团队不该微调"的行业共识,模型级联/路由(快慢路径分流控成本);
- **多模态应用开发**(新增 → `02_多模态应用开发`):vision/audio API 集成、多模态 RAG 的 describe-then-embed 模式、自建 vs 调用 API 的经济性拐点、文档理解是"最不性感但最赚钱"的应用。

### L4 产品与治理层(完全空白,本次新增)

- **AI 产品 UX 设计模式**(新增 → `03_AI产品UX设计模式`):信任校准(不是让用户无脑信任,是让用户的信任程度匹配系统真实能力)、流式响应作为最便宜的信任机制、人机协作断点的交互设计(和《企业落地十大难题》难题⑨呼应,但这里是 UX 设计角度而非架构角度);
- **安全与治理**(现有第 5 关 MCP 安全是子集,需要在意识层面扩展到全链路):提示注入防御、输出过滤、PII 处理、OWASP LLM Top 10——这块建议教师在第 7 关红队环节现场扩展引用本文档,不必单独开新文件。
