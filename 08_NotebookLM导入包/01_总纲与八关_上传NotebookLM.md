# AI 上岗实战训练营 · 总纲(2026 版)

## 课程定位

面向本科生与硕士生的**上岗型训练营**:不从理论教起(基础由 NotebookLM 预习引擎承担),课堂只做三件事——**做项目、被质询、攒证据**。

> 一句话目标:结营时,每个学生有一个**能演示、有评测、有成本记录、有复盘**的 GitHub 作品集,和一套**经过模拟面试压力测试**的表达。

## 为什么是这八关(对标 2026 招聘市场)

2026 年 AI 应用岗的筛选逻辑已经变了:
- 面试第一题不再是"你会调 API 吗",而是"**讲一个你设计过的评测**"
- JD 标配:MCP 集成、Agent 编排、RAG 工程化(切片/召回/重排)、Prompt 版本管理、可观测性、内容安全
- 国内校招 JD 直接写明:"深度使用 Claude Code、Cursor 等 AI 编程工具""设计可复用 Skill""搭建多 Agent 协作系统"
- 简历筛选标准:GitHub 上"干净 repo + eval traces + 成本看板 + 失败复盘" >> 花哨 Demo;开源参与/高星项目可直通二面

## 三舱(每周循环)

| 舱 | 干什么 | 载体 |
|---|---|---|
| 实战舱 | 每周做出可运行的东西 | 课堂 + Claude Code/Cursor |
| 面试舱 | 教练智能体按 4 种面试官人格质询 | 教练智能体 + NotebookLM Debate 模式 |

# 第 1 关 · 侦察关(W1–2)

## 目标
搞清楚两件事:市场在造什么(产品),市场在买什么(岗位)。产出你的**个人赛道声明**——后面七关的所有项目都为这条赛道服务。

## 任务
1. **拆 3 个 AI 产品**(至少 1 个国产、1 个 Agent 形态,如:Kimi/豆包、Cursor、Manus、NotebookLM):用户是谁 / 解决什么 / 技术路线猜测(模型+RAG?+工具?)/ 收费逻辑 / 我能复刻的最小切片
2. **拆 10 份真实 JD**(BOSS直聘/国家大学生就业平台搜"AI 应用开发/大模型应用/Agent 工程师"):把技能要求逐条抄进表格,统计词频
3. **写个人赛道声明**(半页):我瞄准的岗位 × 行业场景 × 我的差异化(专业背景/数据资源/兴趣)

## 验收 rubric
- [ ] JD 词频表里出现:MCP / 评测 / RAG / Agent / 可观测(说明你找的是真 JD 不是老 JD)
- [ ] 每个产品拆解含"我能复刻的最小切片",且切片 ≤2 周工作量
- [ ] 赛道声明能回答:"为什么是你做这个方向?"

## 简历句模板
> (本关不直接产简历句,产**简历的靶心**——之后每关的简历句都朝它写)

## 面试追问预演
- 你为什么选这个方向?(答不出差异化 = 重写声明)
- 这个产品如果让你做,你先砍掉什么功能?

## 双工具用法
- **NotebookLM**:把 10 份 JD 全文上传 → 生成报告"共性技能要求 Top10" → 生成测验自查认知盲区
- **Obsidian**:建 `赛道/` 目录,JD 表格用 Bases 建表(属性:公司/岗位/技能词/薪资),自动聚合词频

# 第 2 关 · AI 编程协作关(W3–4)【v2 新增】

## 目标
把 AI 编程智能体(Claude Code / Cursor / Trae)用成**结对工程师**而不是补全玩具:学会规格驱动开发(spec-driven),学会审查 AI 产出,学会用 git 兜底。

## JD 原文对标
> "搭建多 Agent 协作系统,设计可复用 Skill,**深度使用 Claude Code、Cursor 等 AI 编程工具**"——2026 届国内校招 AI 全栈工程师 JD

## 技术栈
Claude Code 或 Cursor(任选)、git、一门你熟的语言。

## 任务
选一个 200–500 行规模的小工具(建议服务你的赛道,如:JD 爬取整理器、面试题抽认卡生成器),按以下流程做:
1. **写 spec**:先用自然语言写清楚输入/输出/边界条件/验收标准(≤1 页),存进 repo
2. **让 AI 按 spec 实现**:一次一个模块,每个模块你必须读懂再合并
3. **让 AI 写测试**:你负责设计"AI 想不到的刁钻用例"至少 3 个
4. **制造一次事故**:故意让 AI 改坏一个功能,用 git + 测试定位并回滚
5. **复盘**:哪些活 AI 干得好/哪些必须自己来,写 500 字进作战室

## 验收 rubric
- [ ] repo 里有 spec.md,且代码结构与 spec 对应
- [ ] 测试覆盖主要路径,含 3 个自设计刁钻用例
- [ ] git 历史干净(≥8 次有意义提交),含一次回滚记录
- [ ] 能口头解释任意一段 AI 写的代码(抽查)

## 简历句模板
> 熟练使用 Claude Code/Cursor 进行规格驱动开发,独立完成 XX 工具(含测试与 CI),具备 AI 产出代码的审查与回滚能力

## 面试追问预演
- AI 写的代码你怎么保证质量?(答案必须含:spec 约束、测试、审查、git)
- 什么时候你会不用 AI 直接自己写?

## 双工具用法
- **NotebookLM**:上传 Claude Code/Cursor 官方文档 → 音频概览当通勤预习;遇到卡壳先问笔记本再问人
- **Obsidian**:建 `复盘/` 目录,本关复盘是第一篇;以后每关一篇——这个目录期末会变成面试题材库

# 第 3 关 · Prompt 资产关(W5–6)

## 目标
做一个真的有人愿意用的 AI 小产品,并把 Prompt 从"一次性咒语"变成**有版本、有评测的工程资产**。

## JD 原文对标
> "建设 **Prompt 模板与版本管理**、评测集与自动化回归……等应用层通用能力"——2026 届 LLM 应用开发工程师 JD

## 技术栈
任一模型 API、结构化输出(JSON Schema)、git 管理 prompt 模板。

## 任务
1. 从你的赛道里选一个单点任务(如:评论情感抽取、合同条款提取、教案生成)
2. Prompt 写成**模板文件**进 git:角色/任务/约束/few-shot 示例分离,变量用占位符
3. 输出必须是**结构化 JSON**(schema 校验,不是"求模型输出 JSON")
4. 建 20 条评测样本(输入+期望输出),写脚本跑准确率
5. 至少迭代 3 个版本,每版记录:改了什么 → 准确率变化(这就是最小的 prompt 回归)

## 验收 rubric
- [ ] prompt 是文件不是截图,git 里 ≥3 个版本
- [ ] 输出 100% 通过 schema 校验
- [ ] 评测脚本可一键复跑,3 版准确率有记录(如 65%→80%→87%)
- [ ] 有 1 个"改了反而变差"的版本及原因分析(没有 = 迭代不够)

## 简历句模板
> 构建 XX 结构化抽取产品:Prompt 模板化+版本管理,自建 20 条评测集驱动迭代,准确率 X%→Y%

## 面试追问预演
- 为什么这里用 few-shot 而不是 fine-tune?
- 模型升级了你的 prompt 全要重调吗?(考:评测集就是护城河)

## 双工具用法
- **NotebookLM**:上传你的 3 版 prompt + 评测记录 → 用 **Critique 模式音频**听 AI 批评你的设计 → 记录反驳点
- **Obsidian**:评测结果表用 Bases 记录(属性:版本/准确率/改动点),自动生成迭代曲线素材

# 第 4 关 · RAG 工程关(W7–8)

## 目标
做一个**带引用**的知识库问答系统,重点不是"接了向量库",而是**检索质量的工程化**——这是 RAG 岗位面试的分水岭。

## JD 原文对标
> "负责 RAG 链路的工程化落地:**文档接入、清洗与切片、索引构建、召回与重排策略、提示词组装与上下文管理**"——2026 届校招 JD 原文

## 技术栈
任一向量库(Chroma/Milvus)+ BM25 混合检索 + 重排(rerank 模型或 LLM 重排)+ 引用标注。

## 任务
1. 选真实语料(你的赛道领域,≥50 篇文档;禁止用玩具数据)
2. 做两版对比:**V1 朴素版**(固定切片+纯向量)→ **V2 工程版**(语义切片+混合检索+重排)
3. 建 30 条检索测试集(问题+应命中的文档),测**召回率@5**
4. 回答必须带引用,引用错误算失败
5. 长文档陷阱实验:把关键信息埋在文档中部,验证"大海捞针"效应并给出对策

## 验收 rubric
- [ ] V1→V2 召回率有量化对比(如 62%→88%),测试集可复跑
- [ ] 能解释:为什么这个语料这样切片?重排带来多少提升、多少延迟?
- [ ] 10 个抽查回答,引用准确 ≥9
- [ ] 有一条"检索对了但回答错了"的 case 分析(检索≠生成)

## 简历句模板
> 构建 XX 领域 RAG 问答系统:语义切片+混合检索+重排,自建 30 条测试集,召回率@5 从 X% 提升至 Y%,回答全量带引用

## 面试追问预演
- 为什么不直接把文档全塞进 1M 上下文?(考:长上下文让朴素 RAG 更糟,成本/延迟/中部遗忘)
- BM25 和向量检索各自擅长什么?什么时候用 ColBERT/交叉编码器重排?

## 双工具用法
- **NotebookLM**:它本身就是最好的 RAG 竞品——用同一批语料建笔记本,和你的系统**对答 10 题打擂台**,输了的题就是你的优化方向(这个对比写进复盘,面试官超爱)
- **Obsidian**:检索实验记录用 Bases 表(属性:策略/召回率/延迟/成本),Excalidraw 画检索链路图

# 第 5 关 · 工具与 MCP 关(W9–10)【v2 重构】

## 目标
从"会 function calling"升级到"**会造 MCP server**"——让任意 AI 客户端能操作你的真实系统。MCP 已是 2026 年 JD 标配词。

## JD 原文对标
> "实现 MCP servers/clients,跨数据库、SaaS 平台与内部服务编排 API 调用"——2026 年 Agent 工程师 JD;"没有扎实的工具集成,Agent 只是聊天机器人"

## 技术栈
MCP 官方 SDK(Python/TS)、function calling 基础、你的 Obsidian 库(推荐标的)。

## 任务
**推荐路线(创新点):把你的 Obsidian 求职作战室做成 MCP server**
1. 热身:用 function calling 让模型调用 2 个本地函数(查天气/算数),看懂"模型只输出意图,执行在你手里"
2. 用 MCP SDK 实现 server,暴露 3 个工具:`search_notes`(搜作战室笔记)、`add_interview_question`(往错题库写入)、`get_progress`(读关卡进度)
3. 接入 Claude Desktop / Claude Code 实测:对话中让 AI 直接读写你的库
4. **安全加固**:路径穿越防护(禁止访问库外文件)、写操作白名单、危险操作需确认
5. 备选标的:接学校教务数据(自爬)、接小组项目数据库——必须是**真实系统**

## 验收 rubric
- [ ] MCP server 可被至少 1 个真实客户端(Claude Desktop/Code)连接使用
- [ ] 3 个工具各有清晰 description(考:工具描述质量决定模型会不会用)
- [ ] 现场演示一次提示注入攻击被你的白名单/确认机制拦下
- [ ] README 含架构图 + 安装步骤,同学能 10 分钟跑起来

## 简历句模板
> 设计并实现 MCP server(X 个工具,接入真实 XX 系统),含路径防护与操作白名单,已被 Claude Desktop/Code 实际调用

## 面试追问预演
- MCP 和直接 function calling 比,解决了什么问题?(考:M×N→M+N,生态复用)
- AI 通过你的工具删库怎么办?(考:最小权限、确认机制、审计日志)

## 双工具用法
- **Obsidian**:你的库既是开发标的又是文档仓库——**这关做完,你的作战室就是一个能被 AI 操作的活系统,Demo 本身就是面试爆点**
- **NotebookLM**:上传 MCP 规范文档 → Debate 模式生成"MCP 是不是过度设计"正反辩论,听完你才真的懂它

# 第 6 关 · Agent 系统关(W11–12)

## 目标
做一个能**规划、执行、失败恢复**的智能体。区分度不在"能跑",在**失败处理与边界设计**——这是中级和高级候选人的分水岭面试题。

## JD 原文对标
> "基于 LLM 开发具备自主决策能力的 AI Agent""长时运行 Agent 的记忆架构(情景/语义/工作记忆)""多 Agent 协作系统(MCP + A2A)"——2026 年 JD

## 技术栈
模型 API 循环(先裸写 agent loop,再考虑框架)、第 5 关的 MCP 工具、结构化状态存储。

## 任务
1. **裸写 agent loop**(禁止一上来用框架):感知→规划→调工具→观察→循环,≤200 行
2. 接入 ≥3 个工具(复用第 5 关 MCP server)
3. **记忆设计**:任务内工作记忆(上下文管理/摘要压缩)+ 跨会话记忆(文件/DB)
4. **失败处理**(本关灵魂):工具报错重试策略、死循环熔断(最大步数)、卡住时向人求助
5. **护栏**:敏感操作确认、预算上限(token 计数熔断)
6. 【硕士】多 Agent:规划者+执行者+评审者分工,对比单 Agent 的成功率/成本

## 验收 rubric
- [ ] 现场演示:正常完成任务 + **人为注入故障**(断网/工具报错)后优雅恢复或求助
- [ ] 有最大步数与预算熔断,能展示触发记录
- [ ] 日志可回放:每一步的思考/工具调用/观察可查(可观测性)
- [ ] 【硕士】多 Agent vs 单 Agent 有量化对比(成功率/token 成本/延迟)

## 简历句模板
> 开发 XX Agent:自研 agent loop + N 个 MCP 工具,含记忆管理、失败重试与成本熔断,任务成功率 X%(评测集 Y 条)【硕士加:多 Agent 协作使成功率提升 Z%】

## 面试追问预演
- Agent 死循环怎么办?你的熔断阈值怎么定的?
- 为什么不用 LangChain/某框架?(考:知道框架帮你做了什么,才有资格用)
- 你的 Agent 最贵的一次任务花了多少钱?

## 双工具用法
- **NotebookLM**:上传你的 agent 运行日志 → 让它总结失败模式聚类,反哺重试策略设计
- **Obsidian**:Canvas 画 agent 状态机图;失败 case 一条一笔记,标签 `#failure-mode`,期末聚合成"我踩过的坑"——面试终极素材

# 第 7 关 · Evals 上线关(W13–14)【v2 新增 · 硬闸门】

## 目标
给你前面的主项目补齐**生产级验收**:评测、回归、成本、安全。这是 2026 年招聘的第一硬通货——"会评测"是区分"真做过"和"看过视频"的最强信号。**未过此关不得进入发射关。**

## JD 原文对标
> "评测集与自动化回归、A/B 实验、可观测性、内容安全与合规"——国内 JD;"每个候选人都会被要求讲一个自己设计的评测"——海外 2026 招聘指南;评测工程师已成为独立岗位

## 技术栈
pytest 或裸脚本、LLM-as-judge、简单表格即可(不追求平台化)。

## 任务
对你的主项目(RAG 或 Agent):
1. **建评测集 ≥50 条**:含正常 case、边界 case、对抗 case(各占比自定并说明理由)
2. **三种打分器**:精确匹配(能用就用)、程序化校验(schema/引用存在性)、**LLM judge**(写清 judge prompt,并抽 20 条人工标注算与 judge 的一致率)
3. **回归脚本**:一条命令跑全量,输出通过率报告;改 prompt/换模型后重跑对比
4. **成本与延迟表**:每次全量评测的 token 成本、P50/P95 延迟
5. **红队报告**:对自己系统做 ≥5 种攻击(提示注入/越权工具调用/数据带出/角色扮演越狱/长文轰炸),记录攻破与防御
6. 【硕士】judge 与人工一致率 <80% 时,分析分歧模式并改进 judge prompt,写成小实验

## 验收 rubric
- [ ] 评测一键复跑,有 ≥2 次历史报告可对比(证明真的用于回归)
- [ ] LLM judge 与人工标注一致率有数字
- [ ] 成本表能回答:"这系统日活 1000 时每月多少钱?"
- [ ] 红队报告至少 1 个攻破 + 1 个修复前后对比

## 简历句模板
> 为 XX 系统构建 50+ 条评测集与自动化回归(精确匹配/程序校验/LLM judge,judge 与人工一致率 X%),完成 5 类红队测试并修复 Y 个漏洞,建立成本与延迟基线

## 面试追问预演
- 讲一个你设计的评测。(2026 面试第一题,必须 3 分钟讲清:集怎么建、打分器怎么选、发现了什么)
- LLM judge 不可靠怎么办?(考:人工一致性校准、多 judge 投票、分层抽检)
- 你的系统上线后怎么知道它变差了?

## 双工具用法
- **NotebookLM**:上传全部评测报告 → 生成"给面试官的 3 分钟评测故事"提纲;错题(被攻破的 case)生成闪卡反复练
- **Obsidian**:评测历史用 Bases 表(属性:日期/版本/通过率/成本),看板直接展示趋势——截图进简历附件
